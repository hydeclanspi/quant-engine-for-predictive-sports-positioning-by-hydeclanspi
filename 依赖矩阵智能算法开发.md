清楚了。**Resonance（共振评分）不是 correlation（相关性）**，它们是不同的概念。

## Resonance 实际是什么

它本质是 **Dependency Risk Premium（依赖风险溢价）的归一化映射**，映射到 0-100 分：

```
fragilityScore = clamp((adjustedPremium + 0.5) / 1.0 * 100, 0, 100)
```

其中 `adjustedPremium` = 经过幸存者偏差校正和基准率调整后的溢价值。

**直觉解释**：

- **Premium** = `P(AB同时失败|观测) - P(A失败) × P(B失败)`
- 如果两队失败是**独立的**，premium ≈ 0 → score ≈ 50
- 如果两队有**正依赖**（容易同时失败），premium > 0 → score > 50
- 如果两队有**负依赖**（一个失败另一个反而更容易成功），premium < 0 → score < 50

## 和 Correlation 的区别

|          | Resonance                      | Correlation            |
| -------- | ------------------------------ | ---------------------- |
| 衡量什么 | 同时失败的概率是否超过独立预期 | 两个变量的线性共变程度 |
| 数据来源 | 赔率档位匹配的历史组合         | 通常需要连续型数据     |
| 范围     | 0-100（50=独立）               | -1 到 +1               |
| 方法     | P(AB) vs P(A)×P(B) 的差值      | Pearson/Spearman       |

所以更准确地说，Resonance 衡量的是**两场比赛在失败维度上的依赖性超额风险**，"共振"这个词的含义是——如果两场比赛在历史上倾向于"一起崩"（像共振一样），分数就高。







对，理论上完全可能。

Resonance 分数取决于**历史上同赔率档位的组合表现**，不取决于这两场比赛本身的赔率高低。

你举的例子：两场 odds=10 的比赛，resonance=5%，完全合理。意思就是：历史上所有 band=10.0 × band=10.0 的组合里，它们同时失败的概率**低于**独立预期（负溢价），说明这俩高赔比赛在历史上反而不太容易"一起崩"。

反过来也成立：两场 odds=1.2 的大热比赛，resonance 可以很高——如果历史数据显示同档位低赔组合经常一起翻车的话。

**唯一和赔率有关的**是：赔率决定了它落在哪个 band，从而决定去匹配哪些历史数据。但同一个 band 里算出来的分数高低，纯粹取决于历史上那些相似组合的共同失败模式。

## 依赖风险矩阵算法 — 完整管线

### 一、总体目标

对于你当前投注组合中的每一对比赛 (A, B)，计算一个 **Resonance Score（共振评分）0-100**，衡量"这两场比赛同时失败的概率相比独立假设是否偏高"。

核心思想：如果两场比赛在历史上总是一起死，那它们放在同一个组合里就很脆弱（fragile）。

------

### 二、数据源

**历史数据**：从 `localData.js` 取最近 150 条已结算的组合投注，每条包含：

- `matches[]`：每场比赛的赔率 `odds` 和结果 `is_correct`（true/false）
- `succeeded`：整个组合是否盈利
- `createdAt`：投注时间

------

### 三、管线六步走

#### Step 1：赔率分段（Odds Banding）

```
getOddsBand(odds) → band中心值
```

将连续赔率离散化为 8 个档位：

| 赔率范围 | Band |
| -------- | ---- |
| <1.3     | 1.2  |
| 1.3-1.6  | 1.45 |
| 1.6-2.0  | 1.8  |
| 2.0-2.5  | 2.25 |
| 2.5-3.5  | 3.0  |
| 3.5-5.0  | 4.0  |
| 5.0-8.0  | 6.0  |
| ≥8.0     | 10.0 |

**为什么分段？** 不可能找到赔率完全相同的历史对，所以用 band 做模糊匹配。比如当前对是 (1.45, 3.50)，就去找历史上所有 (band=1.45, band=3.0) 的比赛对。

#### Step 2：观测共同失败率（Weighted Observed Co-failure）

```
getWeightedObservedFailure(history, oddsA, oddsB)
→ { failedTogether, totalCount, weightedCount }
```

遍历所有历史组合，在每个组合中找到赔率 band 匹配的一对 (i, j)：

- 如果 `band(i) == band(A)` 且 `band(j) == band(B)`，记为一次观测
- 如果 i 和 j 都 `result === false`（都输了），记为一次共同失败
- 每条记录有**时间权重**：最近 60 天的数据权重 ×1.15，更早的 ×1.0

最终算出：

$$P(\text{both fail} | \text{observed}) = \frac{\text{failedTogether（加权）}}{\text{totalWeightedCount}}$$

#### Step 3：市场隐含失败率（Market Implied）

```
getMarketImpliedFailureRate(odds) = 1 - 1/odds
```

比如赔率 1.45 → 市场隐含胜率 69%，失败率 31%。

然后计算**独立假设下**的共同失败率：

$$P(\text{both fail} | \text{independent}) = P(\text{A fail}) \times P(\text{B fail})$$

#### Step 4：依赖风险溢价（Dependency Risk Premium）

```
calculateDependencyPremium(...)
```

$$\text{Premium} = P(\text{both fail} | \text{observed}) - P(\text{A fail}) \times P(\text{B fail})$$

- **Premium > 0**：历史上共同失败比独立假设更频繁 → 有依赖风险
- **Premium < 0**：历史上共同失败比预期少 → 它们可能有对冲效应
- **Premium ≈ 0**：接近独立

同时计算：

- **样本置信度**：`confidence = ln(1 + sampleSize) / ln(31)`，0→5%，5→52%，10→66%，30+→100%
- **p-value**：二项检验（正态近似），用 erfc 互补误差函数，阈值 0.03

#### Step 5：偏差修正（Bias Correction）

**5a. 幸存者偏差检查** `checkSurvivingBias`

在历史数据中统计这类赔率对出现了多少次：

- **Exact match**：两个维度都精确匹配 band
- **Neighbor match**：至少一个维度是相邻 band（`BANDS` 数组中 index 差 ≤ 1）

如果匹配对 < 3 且总历史 > 10，标记为 `hasBias`（欠采样），对 premium 打折：

$$\text{corrected} = \text{premium} \times (1 - \text{biasStrength} \times 0.5)$$

同时返回 `exactWon` 和 `neighborWon`（都猜对的次数），用于前端展示胜率。

**5b. 基准率调整** `adjustForBaseRate`

用全局失败率做保守调整（conservativeFactor = 0.2）：

- 如果 premium > 0（危险方向）：`adjusted = premium × 0.8`（砍 20%，保守）
- 如果 premium < 0（安全方向）：`adjusted = premium × 1.2`（放大 20%，乐观）

#### Step 6：最终评分（Fragility Score）

```
assessFragilityScore(...)
```

将调整后的 premium 映射到 0-100：

$$\text{score} = \text{clamp}\left(\frac{\text{adjustedPremium} + 0.5}{1.0} \times 100,\ 0,\ 100\right)$$

其中 `maxPremium = 0.5`，所以：

- premium = -0.5 → score = 0（最安全）
- premium = 0 → score = 50（中性）
- premium = +0.5 → score = 100（最脆弱）

**风险等级**：

| Score  | Level             |
| ------ | ----------------- |
| 0-25   | Low               |
| 25-40  | Medium (Moderate) |
| 40-60  | High (Elevated)   |
| 60-100 | Critical          |

------

### 四、组合级聚合

`assessComboFragility` 对 N 场比赛的所有 C(N,2) 对执行上述管线，然后：

- **Overall Fragility** = 所有对的 score 按 confidence 加权平均
- **Critical Pairs** = score > 40 的对，按分数降序排列
- **Recommendations** = 基于 critical pairs 自动生成重组建议

### 方案思路：核密度加权回归（Kernel-Weighted Regression）

核心改造点在 `getWeightedObservedFailure` 和 `calculateDependencyPremium`。

**基本思想**：不再做"匹配/不匹配"的二元判断，而是每个历史对 (h_A, h_B) 对当前对 (A, B) 都有一个**连续的核权重** `K(distance)`：

$$w = K\left(\frac{|bandIdx(h_A) - bandIdx(A)|}{h}\right) \times K\left(\frac{|bandIdx(h_B) - bandIdx(B)|}{h}\right)$$

其中 `K` 可以是高斯核 `exp(-d²/2)`，`h` 是带宽参数。

- 精确匹配：`d=0` → `K=1.0`（满权重）
- 相邻 band：`d=1` → `K≈0.6`
- 隔两个 band：`d=2` → `K≈0.13`
- 很远：`d=5` → `K≈0.00003`（近乎为零但非零）

这样 `(3.0, 6.0)` 的历史数据在计算 `(2.25, 2.25)` 时：

- A维度距离：`|idx(3.0) - idx(2.25)| = 1` → 权重 0.6
- B维度距离：`|idx(6.0) - idx(2.25)| = 3` → 权重 0.01
- 总权重：`0.6 × 0.01 = 0.006`

不高，但非零。如果这种远距离对有很多数据点，它们的累积效应就会体现出来。

2. **有效样本量的重新定义** — 不再是"匹配了几个"，而是所有权重之和 `Σw`，这个值更稳定

### 带宽选择的主流方法

#### 1. 经验法则（Rule of Thumb）

最简单，直接用数据的统计特征算一个"大概合理"的 h。比如 Silverman's rule：

$$h = 0.9 \times \min(\sigma, \frac{IQR}{1.34}) \times n^{-1/5}$$

优点：一行代码搞定，不需要迭代。 缺点：假设数据近似正态分布，如果分布很偏就不太准。

#### 2. 交叉验证（Cross-Validation）

**业界最常用的标准答案。** 思路是：

- 尝试一组候选带宽：h = [0.5, 0.8, 1.0, 1.2, 1.5, 2.0, 3.0]
- 对每个 h，做 leave-one-out：每次拿掉一个历史数据点，用剩下的数据 + 这个 h 来预测被拿掉的那个点的共同失败概率
- 选预测误差最小的那个 h

本质是**让数据自己告诉你最佳平滑度**。不依赖分布假设。

缺点：计算量是 O(|候选h| × N²)，但我们只有 150 条数据，完全可以秒算。

#### 3. 自适应带宽（Adaptive Bandwidth）

不用全局统一的 h，而是**每个区域用不同的 h**：

- 数据密集的区域（比如 odds 1.3-2.5 很多数据）→ 用小 h → 精细刻画
- 数据稀疏的区域（比如 odds 8.0+ 很少）→ 自动用大 h → 从远处借力

实现方式：h(x) = h₀ × (λ / f(x))^α，其中 f(x) 是该位置的数据密度，α 通常取 0.5。

**这个其实最适合我们的场景**，因为低赔率对（1.2-2.5）数据很多、高赔率对（6.0-10.0）数据很少。

#### 4. 贝叶斯方法

把 h 本身也当成一个需要估计的参数，放到贝叶斯框架里，给 h 一个先验分布，然后用 MCMC 或变分推断来得到后验。

优点：理论最优雅，能给出不确定性估计。 缺点：实现复杂度高，对我们这种前端实时计算不太现实。

------

### 我们的场景怎么选？

我推荐 **方法3（自适应带宽）+ 方法2（交叉验证选基础h₀）**：

1. **先用交叉验证**从 [0.8, 1.0, 1.2, 1.5, 2.0] 里选出一个基础 h₀
2. **运行时自适应**：每个 band 位置的局部数据量决定实际带宽

   - 数据少的高赔率区：实际 h 大于 h₀，从邻近借数据
3. **缓存优化**：交叉验证只在历史数据变化时重新跑一次（用户新增结算单时），不是每次渲染都算

这样既有理论基础，又在 150 条数据 + 前端实时渲染的约束下完全可行。

### 方案1：非线性响应函数（Sigmoid/Tanh映射）

**解决的问题**：现在 diff 和修正量是线性关系。比如 diff=0.02 修正 0.02，diff=0.04 修正 0.04，diff=0.10 修正 maxInfluence（截断）。

**用 tanh 映射后**：

```
修正量 = tanh(diff / scale) × maxInfluence × base
```

**业务场景举例**：

假设核回归算出 `rawPremium = -0.08`（平滑估计），最近25场局部算出 `localPremium = -0.03`（局部信号）。

`diff = -0.03 - (-0.08) = +0.05`（局部信号说"没那么负"）

**线性（现在）**：diff=0.05，直接截断到 maxInfluence，修正量 = +0.05 或被 cap 住。

**tanh 映射**：

- 如果 diff 很小（±0.01 量级）→ tanh 接近线性 → 微妙信号被**完整保留甚至略微放大**
- 如果 diff 中等（±0.03）→ tanh 开始弯曲 → 合理修正
- 如果 diff 很大（±0.10，可能是噪声）→ tanh 饱和 → 修正量趋近上限但**不会线性暴增**

本质区别：**线性是"小差异小修正、大差异大修正直到截断"**，tanh 是**"小差异反而相对更敏感，大差异自动柔和压制"**。

### 右上角 CO-FAILURE: "3.785 / 50 pairs failed"

- **50** = 历史上有50场combo里包含了和当前这对赔率相似的配对（raw count）
- **3.785** = 这50场中，**两场都翻车**（result === false && result === false）的加权计数。之所以是小数，是因为每场历史记录都有一个 kernel weight（赔率距离权重）× temporal weight（时间近因权重），所以不是简单的整数计数，而是加权求和
- **"pairs failed"** = 这一对中**两腿都输了**

### 左下角 ODDS PROFILE: "7 hit / 1 won"

- **7 hit** = 历史上有7场combo，里面恰好包含了**完全相同band区间**的一对（Exact match）
- **1 won** = 这7场中，有1场是**这一对两腿都赢了**（`result === true && result === true`）
- **"won" = 这一对两腿都命中**，不是说整个combo全命中

所以 won 和 co-failure 是**镜像关系**：

- won = 两腿都对 ✓✓
- co-failure = 两腿都错 ✗✗
- 中间还有一腿对一腿错的情况 ✓✗，两边都没统计

### 和右下角 CO-FAILURE RATE 的关系

右下角是**概率化**的视角：

- **Observed 16.4%** = `加权co-failure次数 / 加权总次数` = 历史上这对一起翻车的加权概率

### 三者的区别

| 卡片                   | 数据源              | 方法                 | 用途           |
| ---------------------- | ------------------- | -------------------- | -------------- |
| CO-FAILURE (右上)      | 全量历史 kernel加权 | 加权计数             | 原始证据量     |

非常好的问题。`ΔSurvival` 可以理解成：

**“为了满足 A→B 解耦（A翻车时B仍有活路），组合包整体生存率能提升多少个百分点。”**

不是看“相关性好不好看”，而是看**这条关系对你最终不团灭有多大实际价值**。

---

**1) 定义（人话版）**

对某个方向关系 `A→B`：

- `S_before = P(组合包存活 | A翻车)`（当前方案）
- `S_after = P(组合包存活 | A翻车)`（强制满足 A→B 后重算的方案）
- `ΔSurvival(A→B) = S_after - S_before`

`Δ` 越大，说明这条关系越关键，越值得你在矩阵里翻成绿。

---

**2) 具体例子**

假设当前组合包（有资金）是：

- 票1：`A+B`
- 票2：`A+B+C`
- 票3：`A+D`

现在看 `A→B`：

- 当 A 翻车时，含 A 的票几乎都一起死，B 也没有“脱离 A”存在。
- 估计 `S_before = 0.12`（12% 还有活路）

如果强制 `A→B` 后，系统重算成：

- 票1：`A+B`
- 票2：`B+C`
- 票3：`C+D`

则 A 翻车时，`B+C` 和 `C+D` 还能存活，估计：

- `S_after = 0.37`

所以：

- `ΔSurvival(A→B)=0.37-0.12=+0.25`（+25pp）

这就说明：**A→B 是高价值解耦关系**。

---

**3) 为什么它比纯矩阵分数更贴业务**

你业务核心是“防止一场翻车全军覆没”。  
`ΔSurvival` 直接量化了“改这条关系，能多救多少活路”。  
所以它天然比单纯 `premium/fragility` 更接近你的目标函数。

---

**4) 怎么落地到你现在系统（不改架构）**

可以直接在当前框架上做：

- 每条候选关系都算一个 `ΔSurvival`（用现有重算+MC）
- 在冲突弹窗里按 `ΔSurvival` 排推荐项
- 矩阵详情里新增一行：`边际生存增益 +xx pp`

这样你就不是“看起来有点玄学”，而是每次翻转都有明确业务收益解释。
